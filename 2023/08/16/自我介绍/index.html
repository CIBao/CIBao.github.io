# 自我介绍

面试官您好，非常感谢贵公司给予我这次面试的机会。接下来，我想向您介绍一下自己。

我本科就读于福建师范大学，在本科期间，我对算法颇感兴趣，与同学组队参加校 ACM 队，参与过省赛和区域赛，也获得过几次奖项。

而在研究生期间，我的研究方向为云原生和虚拟化方向，特别是 Docker 容器和 Kubernetes 平台。

研一到研二上时，与同门共同研究了容器混合部署场景下的默认驱逐策略导致吞吐量下降的问题，我们针对这些问题进行调研和实验并设计合适的策略来优化问题。

研二下时，参与了华为与我们实验室合作的项目，独自负责研究云场景下多租户资源隔离优化。以上只是我研究中的一些重点，如果您有任何问题，我都会认真回答。再次谢谢。

# 项目介绍

在我们的项目中，我负责研究云场景下容器的资源管理和分配优化这一部分。这是云场景中非常经典的一个问题，大部分工作的优化目标如何保证高优先级任务的性能和提高主机的利用率。而现在最流行的容器管理平台是 k8s，在我的调研中 k8s 侧重于调度，容器网络方面的联系，但是对于容器资源本身的隔离，还是依靠具体容器自身的解决方案。容器最重要点两种机制 namespace 和 cgroup。其中 namespace 负责容器之间互不干扰，使每个容器就像独立的主机，而 cgroup 负责控制容器的资源消耗。而在我们的实验中发现，Cgroup 无法处理容器调用容器外部线程的场景，容器实际上会产生额外资源消耗。举个例子，容器的网络数据包需要结果内核线程 ksoftird（一个专门处理软中断的后台线程）处理，但是内核线程的资源消耗确实全体容器所承担。即使用 Cgroup 设定容器的资源上限为 1 个 CPU，一个恶意容器可以通过不断发送数据包从而实际上可以消耗更多的资源。而且这些外部线程可能导致容器间甚至用户间的资源隔离被打破。针对这个问题，有一篇论文提到过对数据包进行标记并追踪进程在这个数据包的资源消耗最后统计到原有容器的 Cgroup 中，但是，现在并不适用于更通用的场景，因为进程间的调用关系非常复杂，有时候可能通过间接方式比如工作队列，难以追踪。既然每个追踪容器的使用情况比较困难，而且就算实现大概率会变成虚拟机或半虚拟机，额外开销挺大。因此我结合多租户模型，核心思路是为每个用户分配或划分内核线程和后台线程，易于统计和调整。后面的该考虑的问题为每个租户的资源分配问题、和租户内部的资源调整问题。前者有传统的最大最小公平性算法，这个是关注当前资源分配。而基于 Credit 资源分配算法能统计历史资源消耗情况，有人过去用的多，信用就低，要是当下资源不足，优先分配信用高的。对用户分配完资源后，在通过 Prometheus 监控容器的性能指标判断干扰是否发生，利用 cpuset 动态调整 LC 应用和 BE 应用资源范围保证 LC 应用的 QoS。调整 BE 应用的 cpushare 以提高集群资源利用率。
