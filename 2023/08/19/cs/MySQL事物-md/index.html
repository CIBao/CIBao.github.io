# 事物特性

ACID：原子性、一致性、隔离性、持久性

A: undo log

D: redo log

I: MVCC | 锁机制

C: ADI

# 并行事务会引发什么问题？

脏读（读到未提交的值）、不可重复读（数据）、幻读（记录）

# 事务的隔离级别有哪些？

  • 读未提交 （Z、B、H）：直接读最新
  • 读提交 （B、H）：Read View 每个语句执行前
  • ** 可重复读 （H）： **Read View 启动事务时
  • 串行化：用锁

Read View：

  • creator_trx_id ：创建者（事务 id）
  • m_ids: 活跃事件 id 列表 （活跃：启动但未提交）
  • min_trx_id ：活跃事件 id 最小值
  • max_trx_id ：max (m_ids)+1 （代表下一个事件 id）

trx_id: 记录改动的事件 id

roll_pointer：旧版本的记录写到 undo 日志

事件 T 访问记录 R：

  1. T.trx_id < R.min_trx_id : R 可见
  2. T.trx_id >= R.max_trx_id : R 不可见
    1. T.trx in m_ids: R 不可见
    2. T.trx not in m_ids: R 可见

这种通过「版本链」来控制并发事务访问同一个记录时的行为就叫 MVCC（多版本并发控制）

# 可重复读

启动事务时，生成 Read View（只生成一次）

总结：判断记录是否为活跃事件修改，如果是，找旧版本的记录

# 读提交

读数据时，生成 Read View（每次读）

总结：判断记录是否为活跃事件修改，如果是，找旧版本的记录

# MySQL 可重复读隔离级别，完全解决幻读了吗？（无）

快照读：通过 MVCC 方式解决了幻读（在执行第一个查询语句后，会创建一个 Read View，后续的查询语句利用这个 Read View）

当前读：通过 next-key lock（记录锁 + 间隙锁）方式解决了幻读
